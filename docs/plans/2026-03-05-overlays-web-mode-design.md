# castkit Overlays + Web Mode Design (2026-03-05)

## Goal
Add first-class, agent-friendly support for:
- Artifact overlays in terminal demos (images, web snapshots, result cards, charts).
- A new `web` execution/render mode with ScreenStudio-like camera behavior for browser actions.

This design keeps the strict castkit contract (`handoff -> validate -> execute`) and avoids agent guesswork.

## Problem Statement
Current castkit v1 can render polished terminal demos, but it cannot reliably show non-terminal outcomes such as:
- Generated QR code images.
- Website preview screenshots.
- Summarized result cards or charted metrics.

Also, it cannot produce "same vibe" demos for browser workflows (click/type/hover/scroll with zoom focus).

## Design Principles
- Deterministic by default: agents specify artifacts and actions explicitly in schema.
- Evidence-first: new runnable steps/actions still require `source_refs`.
- Backward compatible: existing terminal scripts keep working unchanged.
- Performance-aware: avoid giant payload transfers and full-fps raw captures when unnecessary.
- Minimal cognitive load for agents: small set of high-signal primitives.

## Scope
### In scope
- Step-level overlay artifacts for terminal mode.
- Web screenshot capture utility during execute.
- Web mode schema and runner with deterministic action list.
- ScreenStudio-style web renderer with cursor and focus zoom.
- Docs and agent guidance for storytelling quality.

### Out of scope (initial release)
- Freeform exploratory browser recording.
- OCR-based auto-detection of "important" regions.
- Voiceover generation.
- Full BI dashboard/chart grammar beyond simple chart overlays.

## Proposed API Changes

## 1) DemoScript: add mode
Top-level mode with strict enum:
```json
{ "mode": "terminal" }
```
or
```json
{ "mode": "web" }
```
Default when omitted: `terminal`.

## 2) Terminal step artifacts
Add optional `artifacts` array to `ScriptStep`:
```json
{
  "id": "gen_qr",
  "run": "mycli share --qr out/qr.png",
  "source_refs": ["ref_help_0012"],
  "artifacts": [
    {
      "type": "image",
      "path": "out/qr.png",
      "title": "Share QR",
      "position": "top-right",
      "show_ms": 2400
    }
  ]
}
```

Artifact types (v1):
- `image`: show local image file generated by step.
- `web_snapshot`: capture URL into image and show as overlay.
- `result_card`: key-value summary card from explicit fields.
- `chart`: render simple line/bar chart from provided data file.

Shared overlay fields:
- `title`: optional label.
- `position`: `top-left|top-right|bottom-left|bottom-right|center`.
- `show_ms`: display duration.
- `enter`: `fade|slide|scale`.

## 3) Web mode block
For `mode="web"`, add `web` section:
```json
{
  "mode": "web",
  "web": {
    "base_url": "http://127.0.0.1:4173",
    "viewport": { "width": 1440, "height": 900 },
    "actions": [
      { "id": "open", "type": "goto", "url": "/", "source_refs": ["ref_readme_0004"] },
      { "id": "click_cta", "type": "click", "selector": "[data-demo=cta]", "source_refs": ["ref_files_0021"] },
      { "id": "type_name", "type": "type", "selector": "input[name=q]", "text": "castkit", "source_refs": ["ref_files_0025"] },
      { "id": "assert", "type": "assert_text", "text": "Welcome", "source_refs": ["ref_files_0021"] }
    ]
  }
}
```

Web actions (v1):
- `goto`
- `click`
- `type`
- `press`
- `wait_for_selector`
- `wait_ms`
- `assert_text`
- `screenshot` (can also emit overlay)
- `scroll_to`

## Validation Rules

## Terminal mode
- Existing rules remain.
- If `artifacts` present:
  - `image.path` must be relative (sandbox-safe) and non-empty.
  - `show_ms` must be within safe bounds (e.g., 300..10000).
  - `position` must be valid enum.

## Web mode
- `web` block required.
- `web.actions` must be non-empty.
- Every action must include non-empty `source_refs`.
- Actions needing selectors (`click`, `type`, `wait_for_selector`) require `selector`.
- `goto.url` accepts absolute URL or path relative to `base_url`.
- `assert_text.text` non-empty.
- Disallow arbitrary JS execution action in v1.

## Anti-hallucination
- Action types are strict enums.
- No free-form "do what seems right" actions.
- Unknown fields rejected (`deny_unknown_fields` + schema).

## Execute Pipeline Changes

## A) Terminal overlays
1. Execute step command as today.
2. Resolve and capture declared artifacts after successful step.
3. Store artifact events in transcript:
   - `t_ms`, `step_id`, `artifact_type`, `title`, `image_path`, `position`, `show_ms`.
4. Renderer consumes events and animates overlay layer.

## B) Artifact capture backends
- `image`: verify file exists, optionally resize and transcode to PNG/JPEG cache.
- `web_snapshot`: Playwright capture helper:
  - optional `wait_for_selector`
  - optional clip selector
  - output path in sandbox temp
- `chart`: lightweight render helper from JSON/CSV input (first release: line/bar only).

## C) Web mode runner
- New `execute/web_runner.rs` orchestrates Playwright action execution.
- Produces `WebExecutionTranscript` with:
  - action records
  - timestamps
  - cursor path points
  - target bounding boxes
  - optional keyframe screenshots

## Renderer Changes

## Terminal renderer
- Extend manifest with `overlay_events`.
- Add overlay stage in `renderer-runtime/render.mjs`.
- Animate card/image with easing and z-index above terminal.

## Web renderer
- New runtime entry: `renderer-runtime/render-web.mjs`.
- Renders browser content timeline with:
  - smooth cursor path
  - click pulse
  - focus zoom/pan around active element
  - optional scene tags/watermark/avatar branding
- Encode through existing ffmpeg pipe path.

## Performance Strategy
- Keep manifest compact:
  - avoid duplicating raw lines where possible.
  - store minimal per-frame state.
- Use keyframe + interpolation where safe.
- Keep screenshot capture adaptive (event-driven) instead of per-action flood.
- Reuse image cache for repeated overlays.

## Security + Safety
- Preserve sandboxed execution directory.
- Restrict artifact paths to sandbox subtree.
- Web snapshot URLs default to localhost/relative unless explicit allow flag.
- No remote script evaluation features in web mode v1.

## Backward Compatibility
- Existing scripts without `mode` continue as `terminal`.
- Existing step schema remains valid (`artifacts` optional).
- Existing execute flags keep behavior.

## Developer-Facing Files (planned)
- `src/script/types.rs`: mode + artifacts + web action types.
- `src/schema.rs`: schema updates.
- `src/validate/mod.rs`: mode/artifact/web action validation.
- `src/execute/mod.rs`: mode dispatch.
- `src/execute/artifacts.rs` (new): artifact capture.
- `src/execute/web_runner.rs` (new): web action runner.
- `src/execute/transcript.rs`: overlay + web transcript records.
- `src/render/screenstudio.rs`: terminal overlay manifest support.
- `renderer-runtime/render.mjs`: overlay drawing.
- `renderer-runtime/render-web.mjs` (new): web video renderer.
- `README.md`, `AGENTS.md`, `docs/OVERVIEW.md`: docs and examples.

## Rollout Plan
- Phase 1: terminal overlays (`image`, `web_snapshot`, `result_card`) end-to-end.
- Phase 2: web mode MVP (`goto/click/type/assert/screenshot`) with focus zoom.
- Phase 3: chart overlays + advanced web actions (scroll, key combos, waits).

## Success Criteria
- Agent can generate a terminal demo with at least one real artifact overlay (e.g., QR image).
- Agent can generate a web demo with clicks, typing, and zoom focus that renders deterministically.
- Validation catches malformed artifacts/actions before execute.
- Runtime remains stable for large transcripts/manifests.
